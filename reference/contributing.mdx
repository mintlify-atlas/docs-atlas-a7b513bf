Contributions are welcome. This page covers setup, code standards, and the pull request process.

## Reporting Issues

Before opening a new issue:

1. Search [existing issues](https://github.com/Acly/krita-ai-diffusion/issues) to avoid duplicates.
2. Collect log files from the `.logs` subfolder of the plugin's user data directory (`ai_diffusion`). There is a direct link in the plugin's connection settings.
3. Include: OS (Windows/Linux/macOS), Krita version, plugin version, GPU vendor, and the log files.

## Development Setup

### Cloning and Submodules

```sh
git clone https://github.com/Acly/krita-ai-diffusion.git
cd krita-ai-diffusion
git submodule update --init
```

The repository uses submodules for `debugpy` (the debug server used inside Krita's embedded Python).

### Running the Plugin from Source

The recommended approach uses symlinks so Krita loads the plugin directly from the repository:

1. Locate Krita's `pykrita` folder (the same folder where you normally install plugins).
2. Create a symlink from `pykrita/ai_diffusion` to the repository's `ai_diffusion/` folder.
3. Create a symlink from `pykrita/ai_diffusion.desktop` to `ai_diffusion.desktop` in the repository root.

<Tabs>
  <Tab title="Linux / macOS">
    ```sh
    ln -s /path/to/repo/ai_diffusion ~/.local/share/krita/pykrita/ai_diffusion
    ln -s /path/to/repo/ai_diffusion.desktop ~/.local/share/krita/pykrita/ai_diffusion.desktop
    ```
  </Tab>
  <Tab title="Windows">
    Run Command Prompt as Administrator:
    ```bat
    mklink /D "%APPDATA%\krita\pykrita\ai_diffusion" "C:\path\to\repo\ai_diffusion"
    mklink "%APPDATA%\krita\pykrita\ai_diffusion.desktop" "C:\path\to\repo\ai_diffusion.desktop"
    ```
  </Tab>
</Tabs>

Restart Krita after creating the symlinks.

## Code Style

### Formatting

The project uses [ruff](https://docs.astral.sh/ruff/) for linting and formatting. Run from the repository root:

```sh
ruff format
ruff check
```

Ruff is also run by CI on every pull request. Ensure there are no reported issues before submitting.

### Style Rules

- Follows standard Python style recommendations (PEP 8).
- Exception: no `ALL_CAPS` for constants.

### Type Checking

Type annotations are required where types cannot be inferred. The project uses [pyright](https://github.com/microsoft/pyright) for static type analysis:

```sh
pyright
```

Pyright runs as part of CI. The codebase must report no errors.

<Note>
The `Krita` module is only available when running inside Krita. A type stub file for it is located in `scripts/typeshed/` to allow type checking outside of Krita.
</Note>

## Debugging

The repository includes a `launch.json` for VSCode configured to attach to a running Krita process.

### How It Works

1. `debugpy` is included as a git submodule in the `ai_diffusion/` folder.
2. `extension.py` starts a `debugpy` debug server on startup if the module is present (skipped in release builds).
3. In VSCode, open the "Run and Debug" panel (F5) and select the attach configuration.

### Manual Breakpoints

You can add programmatic breakpoints anywhere in the plugin code:

```python
import debugpy
debugpy.breakpoint()
```

### Workflow Debugging

To inspect the ComfyUI workflow JSON generated by the plugin, enable `debug_dump_workflow` in settings. The exported `workflow.json` is written to the `.logs` folder.

## Running Tests

### Install Test Dependencies

```sh
pip install -r requirements.txt
```

### Run Tests

```sh
pytest tests
```

### What Is Tested

| Area | Tested |
|------|--------|
| Image generation workflows | Yes (limited, due to time cost) |
| Resolution computation | Yes |
| API serialization/deserialization | Yes |
| Krita API integration | No (requires running Krita) |
| UI components | No |

<Note>
Some tests require a running ComfyUI server. These are not yet automated.
</Note>

### Testing the Installer

Installer testing involves large file downloads. To speed this up, pre-download the minimum required models:

```sh
python scripts/download_models.py --minimal scripts/downloads
```

Then run the automated installer tests with a local file server:

```sh
pytest tests/test_server.py --test-install -vs
```

You can also start the file server manually and point Krita at it using the `HOSTMAP` environment variable:

```sh
python scripts/file_server.py
HOSTMAP=1 /your/krita/install/krita
```

## Translations

Translation files are in `ai_diffusion/language/`. Each language has its own JSON file named by language code (e.g., `en.json`, `de.json`).

### Creating a New Translation

1. Copy `new_language.json.template` and rename it to `<language-code>.json`.
2. Fill in the `"id"` and `"name"` fields.
3. Provide translations for each English string in the `"translations"` object.

Example (German):

```json
{
  "id": "de",
  "name": "Deutsch",
  "translations": {
    "Active": "Aktiv",
    "Add Content": "Inhalte hinzufügen"
  }
}
```

### Updating an Existing Translation

After the plugin is updated, new strings are added with `null` values. Search for `null` entries and provide translations.

<Warning>
`{placeholders}` in strings must not be translated or modified — they are replaced at runtime with actual values.
</Warning>

The new language file appears in the plugin's language settings after restarting Krita. You can check the [current translations on GitHub](https://github.com/Acly/krita-ai-diffusion/tree/main/ai_diffusion/language).

## Pull Requests

- For significant changes, open an issue first to discuss the proposal before investing time in implementation.
- Submit changes via a [pull request](https://github.com/Acly/krita-ai-diffusion/pulls).
- Ensure `ruff format`, `ruff check`, and `pyright` all pass before submitting.
- Include relevant tests for new functionality where feasible.
